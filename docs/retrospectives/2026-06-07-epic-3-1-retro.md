## What Could Improve

### Issue 1: Software-Engineer Agent Hit Session Limit Mid-Task (scope: kit)

**What happened:** The first software-engineer agent session completed Tasks 1–6 of 9, then hit the session token limit. Tasks 7–9 (WristbandPDFGenerator, CatalogPDFGenerator, PlayLibraryView, quick-export wiring) were not implemented. The Xcode project file (`project.pbxproj`) already referenced the missing source files, causing immediate build failures.

The orchestrator detected the gap via git log and file existence checks, manually created the 5 missing source files using the plan's embedded code, and verified via test run.

**Root cause:** The implementation plan had 11 tasks totaling approximately 600 lines of source code across 10 new/modified files. This is near or above the reliable completion boundary for a single software-engineer session.

**Action item (scope: kit):** Add a plan-authoring heuristic to CLAUDE.md: plans with more than 6 dense implementation tasks should be split into two sequential plans. Deferred to Ken for exact wording.

---

### Issue 2: `@MainActor` Isolation Errors in Generated Test Files (scope: project)

**What happened:** The software-engineer agent generated `PlayLibraryStoreTests.swift` and `LibraryPersistenceIntegrationTests.swift` without `@MainActor` on the test class, despite calling methods on `PlayLibraryStore` — a `@MainActor final class`. This produced Swift concurrency compile errors. The orchestrator fixed both files with a one-line annotation before SDET ran.

**Action item (scope: project):** Added to `.claude/rules/project-process.md` in this commit. Any `XCTestCase` class that exercises an `@MainActor`-isolated type must be annotated `@MainActor`.

---

### Issue 3: Plan Checkboxes Never Updated (scope: kit)

**What happened:** CLAUDE.md Norm 12 requires implementing agents to update plan checkboxes as each task completes. The software-engineer agent completed Tasks 1–6 but never updated any checkboxes to `[x]`. When the orchestrator needed to assess session 1's progress, plan checkboxes provided no signal.

**Action item (scope: kit):** Add an orchestrator verification step: after accepting an agent's task completion claim, verify the corresponding plan checkboxes are updated before moving to the next task. Deferred to Ken for exact wording.

## Lessons for Future Epics

1. **Plan size predicts session risk.** More than 6 dense tasks in a single plan will likely require multiple sessions or a split plan.
2. **`@MainActor` annotation is non-optional for ObservableObject test subjects.** Pattern is now documented in project-process.md.
3. **Checkbox state is the plan's state-recovery contract.** Unchecked boxes after partial completion are worse than no signal — enforce at task acceptance.
4. **Security-by-design works.** REQ-SEC requirements specified in design, built into plan, and verified post-implementation produced a clean review with no surprises.
